## Must Fix

### 1. Eval feedback is not fragment-specific

Files:

- `scripts/evaluate.ts`

Problem:

- `--feedback` computes one reputation score across the entire evaluated batch.
- The current task set mixes `aquaculture` and `saas-engineering`.
- The on-chain tag is derived from the first task ID prefix rather than a true domain or fragment identifier.

Why it matters:

- The resulting reputation event is ambiguous.
- It can reflect multiple fragments, multiple domains, and multiple task outcomes while looking like one clean score.
- This weakens the core claim that reputation is empirical and attributable.

Recommended direction:

- Submit feedback per fragment or per domain-specific eval batch.
- Include fragment identifiers or content hashes in the evidence path.

### 2. Eval-to-reputation is not bound to a real borrow

Files:

- `scripts/evaluate.ts`
- `scripts/borrow-fragment.ts`

Problem:

- Evaluation loads local markdown from `fragments/`.
- Borrowing fetches content from chain/HTTP and prints it to stdout, but does not persist a borrow artifact that eval consumes.

Why it matters:

- Reputation can be awarded for local files that were never borrowed.
- Reputation can also be awarded for files that were never published or were modified after publish.
- This breaks the integrity of the receipt -> borrow -> eval -> reputation story.

Recommended direction:

- Make eval consume a borrow artifact that includes `fragmentId`, `contentHash`, `contentURI`, and borrowed bytes.
- Refuse `--feedback` unless the evaluated fragment is tied to a specific borrow.

### 3. Checked-in Sepolia config no longer matches the deployed demo

Files:

- `.env.example`
- `contracts/script/Deploy.s.sol`
- `contracts/broadcast/Deploy.s.sol/11155111/run-latest.json`

Problem:

- The repo now deploys mock identity and reputation registries on Sepolia.
- `.env.example` still points to the old `0x8004...` addresses as if those were the active Sepolia registries.

Why it matters:

- Fresh repro attempts are likely to target the wrong contracts.
- The checked-in setup story no longer matches the deployed demo story.

Recommended direction:

- Update `.env.example` to describe the mock-registry deployment path explicitly.
- Consider emitting a machine-readable deploy output for the scripts to consume.

## Okay For Demo

These choices are defensible for a hackathon demo if they are stated plainly:

- Off-chain fragment content with on-chain hashes and receipts
- Sepolia deployment instead of a production-cost chain
- Mock ERC-8004 registries on Sepolia
- Local HTTP fragment serving
- Manual or operator-mediated privacy review

The architecture is strongest where it stays simple:

- `MemoryLending.sol` handles publishing, borrowing, and receipts
- fragment bytes remain off-chain
- identity checks happen on-chain
- evaluation and sanitization remain off-chain

## Critical Review

### Intent

The concept got stronger when it narrowed from “lendable memory” to “corrections are the signal.”

This is the right conceptual move:

- `project` memory is usually too private and too local
- `feedback` memory is where transferable operator skill actually shows up
- extraction + sanitization + publishing is a coherent pipeline for a demo

### Outcome

The current implementation demonstrates a narrower claim than the concept writeup sometimes implies.

What the demo does show:

- correction-like fragments can improve some task responses
- the system can publish, borrow, and emit permanent on-chain receipts
- eval outputs can be transformed into a reputation-shaped signal

What it does not yet show cleanly:

- fragment-level empirical reputation
- reliable improvement across tasks
- true market mechanics for private knowledge
- live ERC-8004 interoperability on Sepolia

Important nuance from the checked-in eval evidence:

- overall delta is positive
- at least one task regressed materially

So the honest claim is:

- borrowed correction context can help
- the protocol flow can be demonstrated

Not yet:

- borrowed memory reliably improves agents

### Architecture

Strongest parts:

- good separation between off-chain content and on-chain attribution
- fragment format remains flexible because the contract stores hash + URI, not the content schema
- the new extraction/sanitization scripts align with the product thesis better than the original generic fragment flow

Weakest parts:

- retrieval/discovery is still primitive
- eval is not tightly coupled to borrow
- privacy safety depends more on process than system enforcement
- reputation aggregation is too coarse to be trusted as a durable marketplace signal

## Post-Hackathon Priorities

1. Bind evaluation to a borrowed fragment artifact.
2. Move from contributor-level batch feedback to fragment-level or domain-scoped feedback.
3. Replace ad hoc fragment discovery with a small catalog/index layer.
4. Make privacy review a required publish-stage workflow, not just a recommended operator behavior.
5. Decide whether production intent is:
   - public tip-jar information goods
   - controlled-access paid fragments
   - or both via different serving paths

## Open Questions

- Is reputation meant to attach to a contributor, a fragment, or a contributor-in-domain?
- Is the long-term product specifically about Claude feedback memories, or about correction fragments from any agent system?
- Should the demo pitch say “ERC-8004-shaped mock registries on Sepolia” explicitly?
- Is verify-before-pay acceptable as the intended economic model, or only as a demo shortcut?
- Do you want review approval to be mandatory before publication of sanitized fragments?
